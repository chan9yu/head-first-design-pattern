# 옵저버 패턴
#스터디/디자인패턴


## 옵저버 패턴의 근본적 문제의식과 핵심 철학
> “의존적 변경의 전파”

- 하나의 상태가 바뀌면 그 상태에 의존하는 여러 객체들도 함께 바뀌어야 하는 상황이 끊임없이 발생
- 이런 “하나가 변하면 여럿이 따라 변해야 한다”는 본질적인 문제에 대한 구조적 대응이 “옵저버 패턴”
- 즉, 데이터를 가진 주체와 그 데이터를 사용하는 관찰자가 서로를 최소한으로만 알면서도, 상태 변경이 자동으로 전파되는 구조를 만드는 것

### 옵저버 패턴이 없다면?
-> 사용자 쪽에서 주기적으로 데이터를 확인

```typescript
// 폴링 방식: 데이터를 사용하는 쪽이 계속 물어봄
class CurrentConditionsDisplay {
  update() {
    const temp = weatherData.getTemperature();
    const humidity = weatherData.getHumidity();
    this.render(temp, humidity);
  }
}

// 어딘가에서 주기적으로 호출해야 함
setInterval(() => {
  currentDisplay.update();
  statisticsDisplay.update();
  forecastDisplay.update();
}, 1000);
```
- 데이터가 바뀌지 않아도 계속 확인 -> 불필요한 연산과 리소스 낭비
- 데이터가 바뀌었지만 다음 확인 주기까지 기다려야 함 -> 실시간성 깨짐
- 새로운 디스플레이가 추가될 때마다 그 디스플레이의 `update()`도 주기적으로 호출하는 코드를 추가해야 함


### 더 근본적인 문제 : 데이터 주체가 소비자를 알아야 한다.
폴링을 개선하기 위해 데이터 주체가 알려준다면?

```typescript
class WeatherData {
  private currentDisplay: CurrentConditionsDisplay;
  private statisticsDisplay: StatisticsDisplay;
  private forecastDisplay: ForecastDisplay;

  measurementsChanged() {
    const temp = this.getTemperature();
    const humidity = this.getHumidity();
    const pressure = this.getPressure();

    // WeatherData가 모든 디스플레이를 직접 알고 있다
    this.currentDisplay.update(temp, humidity, pressure);
    this.statisticsDisplay.update(temp, humidity, pressure);
    this.forecastDisplay.update(temp, humidity, pressure);
  }
}
```
동작은 하나, 문제가 많다.
1. WeatherData가 모든 디스플레이의 구체적 타입을 알아야 함
2. 디스플레이를 추가하거나 제거할 때 WeatherData를 수정해야 함
3. 런타임에 디스플레이를 동적으로 추가하거나 제거할 수 없음

> -> 데이터 생산자와 소비자가 너무 강하게 결합되어 있다!! (강결합, tight coupling)


### 핵심 통찰 : “알림의 구조를 일반화 하자”
> “데이터 생산자는 소비자가 ‘누구’인지 알 필요가 없다. ‘있다’는 것만 알면 된다.”

이것을 코드로 구조화하면, 데이터 주체(Subject)는 관찰자(Observer)의 인터페이스만 알면 됨.
-> 추상적인 계약만 알고, 그 계약을 구현하면, 누구든 등록하고 알림 받는 방식
```typescript
// Subject는 Observer가 "누구"인지 모른다.
// "update를 받을 수 있는 무언가"라는 것만 안다.
interface Observer {
  update(data: WeatherData): void;
}

interface Subject {
  registerObserver(o: Observer): void;
  removeObserver(o: Observer): void;
  notifyObservers(): void;
}
```

> ’전략 패턴’이 “행동의 주체가 행동의 구현을 모르게 만드는 것”이라면,
> ’옵저버 패턴’은 “데이터 생산자가 데이터 소비자를 모르게 만드는 것”

### 느슨한 결합의 원칙

> ”서로 상호작용하는 객체 사이에는 가능하면 느슨한 결합을 사용해야 한다”
> 두 객체가 사호작용하지만 서로에 대해 잘 모르는 상태

-> 이것이 옵저버 패턴에서 어떻게 실현되나?

- Subject는 Observer의 구체 구현을 모름
- 새 Observer를 추가해도 Subject를 수정할 필요가 없음 -> OCP
- Subject와 Observer는 독립적으로 재사용할 수 있음
- 한쪽 변경이 다른 쪽에 영향을 주지 않음

-> 느슨한 결합은 좋은 습관을 넘어 변경에 대응할 수 있는 구조적 안전장치


### Push vs Pull : 데이터 전달 방식의 두 가지 철학
옵저버 패턴의 구현에 있어 중요한 설계 결정 포인트가 있음.

> Subject가 변경을 통지할 때 데이터를 함께 밀어 넣을 것인가? (Push)
> 아니면, 변경 사실만 알리고 Observer가 필요한 데이터를 직접 가져가게 할 것인가? (Pull)

- 전자는 Subject가 모든 데이터를 인자로 넘김 
  -> Observer가 필요하지 않은 데이터까지 받아야하고, Subject의 데이터 구조가 바뀌면 모든 Observer에서 수정이 필요할 수 있음
- 후자는 Subject 자체를 넘기거나 사실만 알림

Pull 모델 예시
```typescript
interface Observer {
  update(subject: Subject): void;
}

class CurrentConditionsDisplay implements Observer {
  update(subject: Subject) {
    if (subject instanceof WeatherData) {
      // 필요한 데이터만 직접 가져감
      const temp = subject.getTemperature();
      this.render(temp);
    }
  }
}
```
-> Pull 모델은 Observer가 자신에게 필요한 데이터만 선택적으로 가져감.


> Push/Pull의 구분은 단순 구현 디테일을 넘어 “변경 통지에 얼마나 많은 정보를 담을 것인가”라는 설계 철학의 문제
